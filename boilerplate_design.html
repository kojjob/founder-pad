Section 01

Top-Level Project Structure

A standard Phoenix app — not an umbrella — with disciplined internal layout. Contexts are the architectural unit. Ash resources live inside them, never exposed directly to LiveView or API layers.

// Directory layout
my_saas/
├── lib/
│   ├── my_saas/                      # Domain layer
│   │   ├── accounts/                 # Users, Orgs, Memberships, Auth
│   │   │   ├── resources/            # Ash resources
│   │   │   ├── policies/             # Ash policy modules
│   │   │   └── accounts.ex           # Public context API
│   │   ├── billing/                  # Plans, Subscriptions, Usage
│   │   ├── ai/                       # Agents, Conversations, Tools
│   │   ├── notifications/            # Emails, Templates, Events
│   │   └── application.ex
│   └── my_saas_web/                  # Presentation layer
│       ├── live/                     # LiveView modules (per context)
│       ├── api/                      # AshJsonApi + AshGraphql routers
│       ├── components/               # Shared LiveView components
│       └── router.ex
├── priv/repo/migrations/
└── config/
Golden Rule LiveView and API controllers call context functions only — never touch Ash resources or Ecto queries directly. Contexts own orchestration. Ash resources own persistence, validation, and policy.
🏛️

Domain Layer

Pure Elixir — no Phoenix dependency. Contexts expose clean public APIs. Ash resources handle all DB interaction.

🖥️

Web Layer

Phoenix + LiveView for the monolith UI. AshJsonApi + AshGraphql for API consumers. Same contexts, different surfaces.

⚙️

Background Layer

Oban workers for Stripe webhooks, AI job queuing, email delivery. Supervised, retryable, observable.

🗄️

Data Layer

AshPostgres with hybrid multi-tenancy. Row-level by default, schema-per-tenant opt-in per resource.

Section 02

Accounts & Auth Context

The load-bearing context. Everything else depends on it. Organisation is the tenant root — billing, AI quotas, and data isolation all hang off the org, not the individual.

// Resource layout
accounts/
├── resources/
│   ├── user.ex              # AshAuthentication lives here
│   ├── organisation.ex      # Tenant root — owns schema_prefix
│   ├── membership.ex        # User ↔ Org join with :role enum
│   └── token.ex             # AshAuthentication tokens
├── policies/
│   └── membership_policy.ex # Role-based access via Ash policies
└── accounts.ex              # Public API: register/2, invite_member/3

Multi-Tenancy Wiring

  • Row-level default — every tenant-scoped resource carries an organisation_id FK, filtered by Ash policies
  • Schema-per-tenant opt-in — single config flag per resource in AshPostgres, no structural rework
  • Tenant context propagation — Accounts loads current_org from session, passed as Ash tenant context downstream

Auth Stack

AshAuthentication

Registration, login, password reset, magic link. All declarative on the User resource.

OAuth2

Google + GitHub via AshAuthentication.Strategy.OAuth2. Providers toggled in config per deployment.

Sessions

AshAuthentication.Phoenix plug — works transparently with LiveView socket assigns.

Key Design Decision Organisation is the tenant root, not User. A user can belong to many orgs. This makes B2B billing, seat management, and data isolation straightforward from day one.
Section 03

Billing Context

Stripe is the source of truth. The local Subscription resource is a read-optimised mirror. No synchronous Stripe calls in the request path.

// Resource layout
billing/
├── resources/
│   ├── plan.ex              # Stripe Product — seeded from config
│   ├── subscription.ex      # Stripe mirror — synced via webhook
│   └── usage_record.ex      # Per-org metered usage tracking
├── webhooks/
│   └── stripe_handler.ex    # Oban worker — async event processing
└── billing.ex               # Public API: create_subscription/2, record_usage/3

Webhook Flow

Update Subscription

Plan Seeding

Plans are defined in config/plans.exs — not managed in the UI. A Mix task syncs them to Stripe and seeds the local DB. Clients never create plans through the app.

Ash Policy Bridge

// Feature gating in any resource
policy action_type(:create) do
  authorize_if Billing.has_active_subscription?(actor.organisation)
end
Usage Metering UsageRecord tracks consumption per org per billing period. Billing.record_usage/3 is called from the AI context after each agent run. Oban aggregates and syncs to Stripe metered billing on a schedule.
Section 04

AI Agent Context

BEAM-native agent orchestration. Concurrent agents run as supervised GenServer processes. Pluggable LLM providers. Tool calls are first-class citizens modelled as Ash resources.

// Resource + process layout
ai/
├── resources/
│   ├── agent.ex             # Agent definition — model, tools, system prompt
│   ├── conversation.ex      # Belongs to Org — holds message history
│   ├── message.ex           # role: :user | :assistant | :tool_result
│   └── tool_call.ex         # Tool invocations — auditable, replayable
├── workers/
│   └── agent_runner.ex      # Oban worker — wraps GenServer agent lifecycle
├── providers/
│   ├── anthropic.ex         # Claude — streaming via SSE
│   ├── openai.ex            # GPT-4o — streaming via SSE
│   └── behaviour.ex         # @behaviour — swap providers transparently
└── ai.ex                    # Public API: run_agent/3, stream_response/2

Concurrency Model

Each agent run is an Oban job → spawns a supervised Task → streams tokens back via Phoenix PubSub → LiveView subscribes and renders in real time. The BEAM handles hundreds of concurrent agent runs without thread pool anxiety.

AI.run_agent/3
→
Oban.insert
→
AgentRunner worker
→
Provider.stream/2
→
PubSub broadcast
→
LiveView render

Provider Behaviour

// Pluggable provider contract
@callback complete(messages :: [map()], opts :: keyword()) ::
  {:ok, stream} | {:error, term()}

@callback supports_tools?() :: boolean()

# Config-driven provider selection per agent
config :my_saas, AI,
  default_provider: AI.Providers.Anthropic,
  fallback_provider: AI.Providers.OpenAI
Tool Calls as Resources ToolCall is an Ash resource — every invocation is persisted, queryable, and auditable. This gives you a replay log, cost attribution per org, and a foundation for agent observability without a separate tracing service.
Billing Integration After each agent run completes, AgentRunner calls Billing.record_usage(org, :ai_tokens, count). Token counting is provider-specific but abstracted behind the behaviour — swap Claude for GPT, metering stays intact.
Section 05

Notifications Context

Transactional and notification email via Swoosh. Template-driven, multi-provider, and observable. In-app notifications via Phoenix PubSub for real-time delivery.

// Resource + mailer layout
notifications/
├── resources/
│   ├── notification.ex      # In-app notification — per user, per org
│   └── email_log.ex         # Delivery audit trail — status, provider, timestamps
├── mailers/
│   ├── auth_mailer.ex       # Password reset, magic link, email verify
│   ├── billing_mailer.ex    # Trial ending, payment failed, invoice ready
│   └── agent_mailer.ex      # Agent run complete, error alerts
├── templates/               # HEEx email templates — Tailwind inline styles
└── notifications.ex         # Public API: send_email/2, notify_user/3

Email Stack

Swoosh

Adapter-based mailer. Resend in production, local Mailbox in dev. Swap adapters in config with zero code changes.

Oban Delivery

All emails enqueued as Oban jobs — retryable, observable, never lost on process crash.

EmailLog

Every outbound email persisted as an Ash resource. Queryable delivery audit trail per org.

In-App Notifications

Notification resource + PubSub broadcast. LiveView subscribes on mount — real-time bell icon updates.

Transactional vs Notification Split

Type Mailer Trigger Delivery
Transactional AuthMailer Password reset, magic link Immediate Oban job
Billing BillingMailer Stripe webhook events Chained from StripeHandler worker
Notification AgentMailer AI run complete / failure Oban + PubSub in parallel
Provider Portability The Swoosh adapter is set in config — clients using SendGrid, Postmark, or Resend need zero code changes. The boilerplate defaults to Resend (best DX + deliverability) but ships adapter-agnostic.
Section 06

API Layer — REST + GraphQL

AshJsonApi and AshGraphql derive the entire API surface from Ash resource definitions. Zero hand-written controllers for CRUD. Custom actions map to non-standard endpoints cleanly.

// API router layout
my_saas_web/api/
├── json_api_router.ex       # AshJsonApi — mounted at /api/v1
├── graphql_schema.ex        # AshGraphql — mounted at /api/graphql
└── openapi_spec.ex          # Auto-generated OpenAPI 3.0 spec

Resource → API Mapping

// Ash resource — both APIs derived from one definition
defmodule MySaas.Accounts.User do
  use Ash.Resource,
    domain: MySaas.Accounts,
    extensions: [
      AshJsonApi.Resource,
      AshGraphql.Resource,
      AshPostgres.DataLayer
    ]

  json_api do
    type "user"
    routes do
      base "/users"
      get :read
      index :list
      post :register         # custom action → POST /users/register
    end
  end

  graphql do
    type :user
    queries do
      get :get_user, :read
      list :list_users, :list
    end
    mutations do
      create :register_user, :register
    end
  end
end

Auth on the API Layer

Bearer Tokens

API clients authenticate with Bearer tokens. AshAuthentication issues them — same token infrastructure as the web session.

Tenant Scoping

Every API request resolves org_id from the token. Ash policies enforce tenant isolation automatically — no per-query filtering needed.

OpenAPI Spec

AshJsonApi auto-generates an OpenAPI 3.0 spec at /api/v1/open_api. Rails/Django/Next.js clients can generate typed SDKs from it.

API-Escapable Pattern When a client needs Rails or Django to consume your Phoenix backend: point them at /api/v1 (REST) or /api/graphql. The OpenAPI spec lets them auto-generate client SDKs. Phoenix stays the system of record — the other framework is just a consumer.

Cross-Framework Integration

How Rails, Django, Node.js, or Next.js plug into this boilerplate — without compromising the Phoenix core.

The Integration Boundary

Phoenix / Ash
→
/api/v1 (REST)
→
Rails / Django / Node
Phoenix / Ash
→
/api/graphql
→
Next.js / Apollo
Framework Integration Point What it gets What Phoenix owns
Rails REST /api/v1 Admin panel, reporting, backoffice Auth, data, business logic, billing
Next.js GraphQL /api/graphql Marketing site, SEO frontend Auth tokens, all mutations, AI
Django REST /api/v1 ML pipeline, data processing Auth, multi-tenancy, billing
Node / Express REST or GraphQL Webhook relay, integrations Core domain, persistence
The Rule Other frameworks are consumers of the Phoenix API — never co-owners of the database. Phoenix owns the schema, migrations, and business logic. Rails/Django/Node never write to the DB directly. This prevents the "two migration systems" nightmare.
What You Never Do Never share the PostgreSQL database between Phoenix and another framework at the write level. Rails has its own migration system. Django has its own ORM. Letting two frameworks own the schema simultaneously is the fastest way to corrupt your data model. Read-only access via views or a reporting schema is acceptable for analytics.

Recommended Client Delivery Patterns

🚀

Default Delivery

Full Phoenix monolith. LiveView for UI. No other framework. Fastest to ship, most cohesive. 90% of client projects.

🔌

API + Next.js Frontend

Client insists on React frontend. Phoenix serves GraphQL. Next.js consumes it. LiveView dropped for this deployment.

🔧

Phoenix Core + Rails Admin

Client has existing Rails codebase. Phoenix becomes the AI/real-time engine. Rails consumes REST API for backoffice.
