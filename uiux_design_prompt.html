01

Master Prompt

Use this as your primary generation prompt — for v0, Cursor, Figma AI, or a human designer briefing.

MASTER PROMPT Design a sleek, responsive, professional B2B SaaS application UI for a multi-tenant AI-powered platform built on Phoenix LiveView and TailwindCSS. The product serves technical teams and indie founders who manage AI agents, subscriptions, and organisational workspaces.

Visual Direction: Refined dark-first aesthetic. Deep navy/charcoal backgrounds with crisp white surface cards. A single strong accent colour (electric indigo #6366f1) used sparingly for CTAs, active states, and data highlights. Secondary warm amber accent for billing and usage indicators. Clean geometric sans-serif typography — no decorative fonts. Generous whitespace, precise 8px grid spacing system, subtle depth via layered surface elevations rather than heavy drop shadows.

Layout System: Full-height sidebar navigation (240px collapsed to 64px on mobile) with nested context sections. Top header bar with global search, notification bell, org switcher, and user avatar. Main content area with max-width 1280px container. Responsive breakpoints: mobile ≥ 375px, tablet ≥ 768px, desktop ≥ 1280px. All interactive surfaces must work at every breakpoint.

Component Language: Rounded corners (radius 8px cards, 6px buttons, 4px inputs). Monospace font for all data, IDs, tokens, and code snippets. Subtle 1px borders in dark surface colours rather than heavy outlines. Micro-interactions on hover (100ms ease transitions). Loading states with skeleton screens — no spinners. Empty states with actionable illustration placeholders. Toast notifications anchored bottom-right.

Data Density: Design for information-rich views — dashboards with multiple metric cards, agent run logs with status badges, billing tables with usage bars. Every table must have: sortable column headers, row hover states, inline action menus, pagination, and a "no results" empty state. Prioritise scannability over decoration.

Technical Constraints: All components must be implementable in Phoenix LiveView with TailwindCSS utility classes only — no custom CSS frameworks. Avoid JavaScript-heavy patterns; prefer CSS-driven interactions. Forms use LiveView phx-change and phx-submit patterns — design validation states (error, success, loading) for every form field. Real-time updates (agent streaming, notification counts) must have smooth DOM-patch-friendly designs — no full-page repaints.

04

Per-Screen Prompts

Use each block as a targeted prompt for individual screen generation.

Home Dashboard

SCREEN PROMPT Design the main dashboard screen for a multi-tenant AI SaaS platform. Dark background (#0f1117). Top header with org name, global search bar, notification bell with unread badge, and user avatar dropdown.

Left sidebar (240px) with navigation groups: Overview, Agents, Billing, Team, Settings. Active state uses left border accent + subtle background tint. Collapse to icon-only at tablet breakpoint.

Main content area: 4 metric cards in a row — Active Agents, AI Tokens Used This Month, Team Members, Current Plan. Each card has an icon, large numeric value in monospace, percentage change badge (green/red), and sparkline chart.

Below: Recent Agent Runs table (agent name, status badge, duration, tokens used, triggered by, timestamp) and a Usage Over Time line chart (30 days, token consumption, colour-filled area under the line). Bottom: upcoming billing date notice bar.

AI Agent Live Run

SCREEN PROMPT Design the live agent run screen — the most interactive screen in the product. Split-pane layout: left pane (40%) shows agent config (model, system prompt, tools enabled, temperature). Right pane (60%) is the live output stream.

Output stream: scrollable message thread. User messages in right-aligned bubbles. Assistant messages left-aligned, streaming token by token with a blinking cursor. Tool call blocks styled as collapsible code cards with tool name, input JSON, and result JSON. Status bar at bottom: tokens used, elapsed time, model name.

State variants required: idle (run button prominent), running (animated pulsing status indicator, abort button), complete (summary card with cost, token count, run ID in monospace), error (red alert card with error message and retry button).

All transitions must be LiveView-patch-friendly — no full DOM replacement during streaming. New message tokens append to existing DOM nodes.

Billing Dashboard

SCREEN PROMPT Design the billing and usage dashboard. Top section: current plan card (plan name, price, renewal date, upgrade/downgrade CTA). Status badge: ACTIVE in green, TRIAL in amber, PAST DUE in red.

Usage section: horizontal progress bars for each metered resource (AI tokens, API calls, team seats) — value / limit with colour shift from green → amber → red as limit approaches. Overage warning banner when at 80%+.

Invoice table: date, description, amount, status badge (paid/pending/failed), download PDF link. Stripe-hosted portal link as secondary CTA.

Payment method card: masked card number, expiry, card brand icon, "Update" link. Design a trial countdown card variant: days remaining as a large number, progress ring, upgrade prompt.

Authentication — Sign In

SCREEN PROMPT Design the sign in screen. Full-height split layout: left half is a dark branded panel with product logo, tagline, and a subtle abstract geometric pattern or grid. Right half is the auth form on white/light surface.

Form: email input, password input (show/hide toggle), "Remember me" checkbox, "Forgot password?" link. Primary CTA: "Sign in" button (full width, accent colour). Divider: "or continue with". OAuth buttons: Google and GitHub (outlined, icon + label).

State variants: default, loading (button shows spinner + "Signing in…"), error (inline red error message above submit), success (redirect animation). Footer: "Don't have an account? Sign up" link.

Responsive: stacked single column on mobile, left panel hidden on screens below 768px.

Team Members

SCREEN PROMPT Design the team members management screen. Page header with member count and "Invite Member" primary button (opens a slide-over panel).

Members table: avatar + name, email, role badge (:owner / :admin / :member in distinct colours), joined date, last active timestamp, actions menu (Edit Role, Remove). Current user's row subtly highlighted. Owner row has crown icon, non-removable.

Pending invites section below the table: email, invited by, sent date, "Resend" and "Revoke" actions. Empty state: illustration + "No pending invites".

Invite slide-over panel: email input, role selector (radio group), optional personal message textarea, "Send Invite" CTA. Slide in from right, backdrop blur behind.

  • Primary — filled accent
  • Secondary — outlined

  • Danger — red filled

  • Select dropdown

06

Tool-Specific Tips

v0 by Vercel

Prefix every v0 prompt with: "Using TailwindCSS utility classes only, no custom CSS. Dark theme. shadcn/ui components where applicable." Then paste the screen prompt. v0 responds best to single-screen prompts — do not combine multiple screens in one generation.

Figma AI / Figma Make

Use the Master Prompt as the design system brief first. Then use per-screen prompts as individual frame briefs. Ask Figma to generate a component library from the colour palette and typography scale before generating screens.

Cursor / Claude Artifacts

Prepend: "Generate as a single-file Phoenix LiveView HEEx template using TailwindCSS." For interactive components, append: "Include phx-click, phx-change, and phx-submit bindings where appropriate. All state managed via LiveView assigns."
