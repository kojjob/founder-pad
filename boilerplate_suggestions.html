Audit Log Context
ash_notifier + AuditLog resource
High
›
Every significant action in the system — resource creation, deletion, policy decisions, billing events — persisted as an immutable AuditLog record. Wired in via an Ash notifier so you never forget to add it manually. Queryable per org, per actor, per resource type.
# lib/my_saas/audit/resources/audit_log.ex defmodule MySaas.Audit.AuditLog do use Ash.Resource, domain: MySaas.Audit attributes do attribute :actor_id, :uuid attribute :org_id, :uuid attribute :action, :atom attribute :resource, :string attribute :changes, :map attribute :ip_address, :string create_timestamp :inserted_at end end
GDPR compliance Debugging in prod Enterprise requirement SOC2 audit trail

Feature Flags
FeatureFlag resource, plan-gated
High
›
A FeatureFlag Ash resource scoped per org and per plan tier. Ship features behind a flag and gate them without a code deploy. Plan upgrades become instant — flip the flag, the feature appears. Also enables beta programs: enable a flag for select orgs before general release.
# Usage in any context if FeatureFlags.enabled?(:ai_agents, org) do # render AI agent UI end # Ash policy integration policy action_type(:create) do authorize_if FeatureFlags.enabled?(:agents, actor.org) end
Zero-downtime releases Beta programs Plan gating

API Rate Limiting
Hammer plug, per token + per org
High
›
Essential once you expose AshJsonApi and AshGraphql publicly. Without it, one runaway client can saturate your database. Use Hammer with ETS backend for speed, Redis backend for multi-node. Limits per API token, per org, and globally. Return 429 with Retry-After headers.
# plug in your API router plug MySaasWeb.Plugs.RateLimit, limit: 1000, window_ms: 60_000, key: &extract_token_id/1 # per-plan limits from config @limits %{free: 100, pro: 1000, enterprise: :unlimited}
Abuse prevention DB protection SLA enforcement

Outbound Webhooks
Client-defined endpoints, Stripe-style
High
›
Your platform publishing events to client-defined endpoints. Critical once clients build automations on top of your AI agents — they need to know when a run completes, fails, or produces output. OutboundWebhook resource stores endpoint URLs + secret. Oban handles delivery, retry, and signature verification.
# Events published by the platform agent.run.completed agent.run.failed billing.subscription.activated billing.payment.failed org.member.invited
Client automations Zapier / Make.com Integration layer

Mix Task Suite
setup, sync_plans, seed, reset
Medium
›
A suite of project-specific Mix tasks that make dev and deployment operations predictable. One command per operation. No documentation needed — the task names are self-describing. Critical for client handoffs where the client's dev team needs to bootstrap without your tribal knowledge.
mix my_saas.setup # deps + db + seeds mix my_saas.sync_plans # sync Stripe products → DB mix my_saas.seed # realistic demo data mix my_saas.reset # drop + recreate + seed

Onboarding Flow
Org setup → invite → first run
High Missing
›
The current prompt lists onboarding as a screen but has no spec. It is the most conversion-critical screen in the product. Design as a 4-step wizard: (1) Name your org, (2) Invite your first teammate, (3) Configure your first AI agent, (4) Watch it run. Progress stepper at top. Skip all = straight to dashboard.
Activation rate Time to first value Churn prevention Trial conversion

Dark + Light Mode
Tailwind dark: classes + system pref
High
›
The design prompt specifies dark-first, but never mentions a light mode variant. Many clients' end users expect light mode. Tailwind's dark: prefix handles this with zero JavaScript — respects system preference by default, overridable via a class toggle stored in localStorage. Design both themes and test every component in each.
<!-- tailwind.config.js --> darkMode: 'class' // or 'media' for system only <!-- every surface needs both --> class="bg-gray-900 dark:bg-gray-950 text-gray-100 dark:text-gray-50"
User preference Accessibility Professionalism

API Keys Screen
Create, revoke, scope, copy once
High Missing
›
Not in the current UI prompt at all. Essential for the API-escapable model — Rails/Django/Next.js consumers need API keys to authenticate. Design: table of keys with name, prefix (sk_live_••••••8f2a), scopes, last used, created date. Show full key once on creation with a "Copy" button then never again. Revoke with confirmation modal.
API-escapable model Client integrations Security UX

Settings Architecture
Org vs Personal vs API — split
High
›
The current prompt has a single "Org Settings" screen. This is wrong. Split into three separate settings areas: (1) Org Settings — name, logo, domain, danger zone; (2) Personal Settings — profile, password, notification prefs, connected accounts; (3) API Settings — API keys, webhooks, rate limit info. Mixing them is the most common settings UX anti-pattern.
User mental model Permission clarity Scalability

Empty + Error States
400, 404, 500 + contextual empties
Medium
›
The UI prompt mentions empty states vaguely. Be explicit: every list view needs a bespoke empty state with an actionable CTA ("Create your first agent →"), not just a generic "No results." Design custom 400, 404, 503 error pages consistent with the product's visual language. These are client-facing and often the first screen new users see when something goes wrong.

Mobile-First LiveView
Bottom nav, drawer, touch targets
Medium
›
The current prompt handles responsive breakpoints but doesn't address LiveView-specific mobile patterns. Design a bottom navigation bar for mobile (replaces sidebar), a slide-up drawer pattern for secondary actions, and ensure all touch targets are ≥44px. LiveView's DOM-patch model is mobile-friendly by default — but test every phx-click on touch devices explicitly.

Demo Mode
Seeded read-only login, no signup
High
›
The single highest-ROI addition to the boilerplate. A seeded login (demo@example.com / demo) that lands in a fully-populated read-only environment — realistic org, agents with run history, billing data, team members. Clients click a link, see the full product running with data, and immediately understand what they're buying. No demo mode = longer sales cycles.
# One env variable activates demo mode DEMO_MODE=true # All mutations blocked in demo policy action_type(:create) do forbid_if actor.is_demo_user end
Faster sales cycles Self-serve trials Investor demos

Mix Generator
mix phx.gen.saas — module picker
High
›
Instead of clients manually deleting modules they don't need, ship a Mix generator that lets them pick modules at project creation time. Not every client needs AI agents. Some don't need multi-tenancy. The generator scaffolds only what was selected, keeping the codebase clean from day one. Similar to Rails generators but for your specific boilerplate.
mix phx.gen.saas MyApp \ --modules auth,billing,ai \ --tenancy row_level \ --api rest,graphql
Faster starts Cleaner codebases Client autonomy

Test Scaffolding
ExMachina factories + DataCase
High
›
Ship the test infrastructure as first-class citizens — not an afterthought. ExMachina factories for every resource (User, Org, Membership, Subscription, Agent). A custom DataCase with multi-tenancy helpers: build_org/1, build_member/2. LiveView test helpers for common interactions. This is what makes TDD on a new client project actually possible from day one.
# test/support/factory.ex def org_factory do %Organisation{name: sequence("Org ")} end # Ready to use in every test insert(:org) insert(:member, org: org, role: :admin)
TDD from day 1 CI reliability Client handoff quality

White-Label Config
Logo, brand colour, domain — config only
Medium
›
Every client project you deliver looks different. Make branding a first-class concern in the boilerplate. A config/branding.exs file with logo path, primary colour, product name, and support email. LiveView layouts read from this config — no hardcoded brand anywhere. A client who wants to rebrand changes one file and redeploys.
# config/branding.exs config :my_saas, :branding, product_name: "AcmeSaaS", primary_color: "#6366f1", logo_path: "/images/logo.svg", support_email:"support@acme.com"
Client rebranding White-label delivery Multi-client reuse

Docs Site
ExDoc — module-by-module reference
Medium
›
Your boilerplate's biggest competitor is "I'll just build it myself." Documentation is the counterargument. ExDoc auto-generates a reference site from your module docs. Add a hand-written guides/ folder with: Getting Started, Context Architecture, Adding a Module, Deploying. Ship the docs site at a subdomain of your product site. Clients reference it instead of asking you.

Version + Changelog
GitHub template, semver, CHANGELOG.md
Medium
›
Treat the boilerplate as a versioned product, not a one-time scaffold. Publish it as a GitHub template repository. Tag releases with semver. Maintain a CHANGELOG.md in Keep a Changelog format. Clients who adopted v1.0 can track what changed in v1.2 and selectively apply updates. This also signals seriousness to prospective clients evaluating the boilerplate.
